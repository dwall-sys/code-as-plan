    Step 1 (load_context):
    - Read CLAUDE.md if exists
    - Read .planning/prototype/CODE-INVENTORY.md -- extract all @gsd-api tags as test specifications
    - Read .planning/prototype/PROTOTYPE-LOG.md for source file list
    - Detect test framework: `node "$HOME/.claude/get-shit-done/bin/gsd-tools.cjs" detect-test-framework "$PWD" --raw`
    - Discover existing test directory by globbing for `**/*.test.*` and `**/*.spec.*`
    - If no test directory found, plan to create `tests/` at project root (per D-04)
    - Read arc-standard.md for @gsd-risk annotation syntax

    Step 2 (plan_tests):
    - For each @gsd-api tag: map to test cases (happy path + at least one edge case per contract)
    - Identify @gsd-constraint tags as boundary condition test candidates
    - List planned test files with paths and approximate test count
    - Use framework-appropriate file extension: node:test -> .test.cjs, jest/vitest -> .test.js or .test.ts (match project convention)

    Step 3 (write_tests):
    - Write test files using the Write tool (per D-01 -- never Bash heredoc)
    - Use the framework syntax detected in Step 1 (import/require style, describe/test/it blocks, assertion library)
    - CRITICAL (per D-03): Each test asserts the API CONTRACT from @gsd-api tags, NOT stub return values
    - Example of WRONG test: `assert.strictEqual(result, undefined)` (passes against stubs)
    - Example of RIGHT test: `assert.ok(result.id); assert.strictEqual(typeof result.email, 'string')` (fails against stubs that return undefined/null/{})
    - Test naming follows project's existing convention (discovered in Step 1)

    Step 4 (red_green):
    - RED PHASE: Run tests using the detected testCommand via Bash tool
      - Read FULL Bash output -- look for summary line, not individual test lines (Pitfall 6 from RESEARCH.md)
      - Framework-specific failure signals: node:test -> `# tests failed` or `not ok`; vitest -> `FAIL`; jest -> `FAIL`; mocha -> `failing`; ava -> `not ok`
      - If ALL tests PASS on RED: tests are too weak. Rewrite with stricter contract assertions (per D-08). Do NOT proceed to GREEN.
      - If some tests fail: RED confirmed. Record which tests failed and why.
    - GREEN PHASE: Verify tests pass against the real implementation
      - Run tests again after implementation exists
      - If tests FAIL on GREEN: debug test logic (not implementation). Fix assertions that are incorrect, not overly strict.
      - If tests PASS: GREEN confirmed.
    - IMPORTANT: If the code is still in stub/scaffold state (all functions return undefined, throw 'not implemented', or return hardcoded primitives), only the RED phase applies. Document that GREEN will be confirmed after implementation. Do NOT fake GREEN.

    Step 5 (annotate_risks):
    - Scan source files for code paths NOT covered by generated tests
    - Candidates: complex async flows, external HTTP/DB calls, UI interactions, event handlers with side effects, dynamic imports
    - Add @gsd-risk tag on its own line ABOVE the code path, with comment token as first non-whitespace (per arc-standard.md comment anchor rule)
    - Metadata keys: `reason:` (why untested) and `severity:` (high/medium/low) per D-10
    - Example: `// @gsd-risk(reason:external-http-call, severity:high) sendEmail calls SMTP -- cannot unit test without mocking`
    - Report summary: test file paths created, total test count, @gsd-risk annotations added with locations

    **constraints section**:
    - NEVER write tests that assert stub return values -- always assert the contract from @gsd-api
    - NEVER skip the RED phase -- must confirm tests actually fail before claiming GREEN
    - NEVER place @gsd-risk inline after code -- it must be on its own line with the comment token first
    - ALWAYS use Write tool for file creation, never Bash heredoc
    - ALWAYS read the FULL test output summary line, not individual test results
    - If no @gsd-api tags found in CODE-INVENTORY.md, report this and exit -- there is nothing to test
    - Common stub patterns that tests MUST fail against: `return undefined`, `return null`, `return {}`, `throw new Error('not implemented')`, `return 'TODO'`, hardcoded primitive returns that don't match the @gsd-api contract shape
