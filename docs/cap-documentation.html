01Core Philosophy

Eliminate upfront document-heavy planning. Let your code be the plan.

The Problem

Traditional Plan-Heavy Workflow

  • Weeks spent writing requirements documents
  • PRDs become stale before code is written
  • Traceability lost between plan and code
  • Developers duplicate information
  • Context switching between docs and code

CAP: Prototype-Driven

  • Start building immediately
  • Structure emerges from annotated code
  • Tags are the traceability link
  • Feature Map is auto-enriched from code
  • Everything stays in the codebase

How CAP Is Different

AspectTraditionalCAP
PlanningUpfront documents (PRD, RFC, ADR)Code tags + Feature Map
Source of truthWiki / Google DocsThe codebase itself
TraceabilityManual linkingTag scanner auto-links
StalenessDocs drift from realityTags live in code, always current
OnboardingRead 50 pages of docsRead Feature Map + grep tags

Prototype-Driven Development

CAP implements the "Code-First" principle: build a working prototype first, then extract structured planning artifacts from the annotated code. This inverts the traditional workflow where planning precedes coding.

Aligned with Farley's Principles

Dave Farley's Modern Software Engineering advocates optimizing for learning and managing complexity. CAP operationalizes this by:

  • Optimizing for learning: Brainstorm, prototype fast, test against acceptance criteria, iterate
  • Managing complexity: Feature Map as single source of truth, tag-based traceability, agent separation of concerns

Structure

Each feature entry in FEATURE-MAP.md contains:

  • Feature ID — F-NNN (zero-padded)
  • Title — human-readable name
  • State — lifecycle position
  • Acceptance Criteria — testable conditions (the contract)
  • Tags found — auto-populated by /cap:scan
  • Dependencies — other feature IDs

Example

## F-001 User Authentication
State: prototyped
Group: Core
Dependencies: none

### Acceptance Criteria
- [ ] Users can register with email and password
- [x] Passwords are hashed with bcrypt
- [ ] JWT tokens expire after 24 hours
- [ ] Rate limiting on login endpoint (5/min)

### Tags Found
- src/auth/login.js: @cap-feature, 2x @cap-todo, 1x @cap-decision
- src/auth/register.js: @cap-feature, 1x @cap-todo

06Project Memory System

Persistent, code-derived project knowledge that survives context resets and spans developers.

Overview

Project Memory captures the why behind your code — decisions, risks, patterns, and hotspots — extracted directly from the codebase. Unlike chat-based memory that fades with context windows, CAP memory is git-tracked, team-shared, and code-derived.

Code Tags (Primary)

@cap-decision and @cap-todo risk: tags are the ground truth. Zero noise, 100% signal. Memory is extracted from code, not guessed from conversations.

Session Hotspots

Edit frequency tracking across sessions reveals which files keep changing. Invaluable for onboarding new team members.

Conversation Threads

Brainstorm sessions are persisted as threads with branching, revisit detection, and multi-developer support.

Code-First Memory

Memory entries originate from code annotations, not session heuristics. This is the key differentiator: zero noise, 100% signal.

How a @cap-decision becomes a memory entry

▼

Step 1: Developer writes a decision in code:

// @cap-decision F-001 Using bcrypt over argon2 — broader ecosystem support

Step 2: /cap:memory init runs accumulateFromCode(), which scans all files:

// Tag scanner finds the decision tag
{
  type: "decision",
  feature: "F-001",
  text: "Using bcrypt over argon2 -- broader ecosystem support",
  file: "src/auth/login.js",
  line: 14,
  anchor: "decision-f001-bcrypt-argon2"
}

Step 3: Entry appears in .cap/memory/decisions.md and as a node in the memory graph:

## F-001: Using bcrypt over argon2
Source: src/auth/login.js:14
Reason: Broader ecosystem support
Anchor: decision-f001-bcrypt-argon2

Pitfalls from @cap-todo risk: and @cap-risk

▼

Risk-tagged todos and explicit @cap-risk annotations become pitfall entries in memory. These persist across sessions so new developers see known hazards immediately.

// @cap-todo risk:security F-001 Token rotation not yet implemented
// @cap-risk F-003 Third-party API has 99.5% SLA, need fallback

Both become entries in .cap/memory/pitfalls.md with anchor IDs for deduplication.

Session Hotspots

Hotspots answer the question: "Which files keep getting changed?" This is tracked by analyzing edit frequency across session JSONL files.

FileEditsSessionsLast Edited

Conversation Threads F-031

Brainstorm sessions are persisted as threads in .cap/memory/threads/ with a central thread-index.json.

Thread Persistence

▼

Every /cap:brainstorm session creates a thread file. Threads capture:

  • Feature IDs discussed
  • Acceptance criteria proposed
  • Key decisions made
  • Open questions remaining
  • Timestamp and developer identity
// .cap/memory/threads/thread-2024-03-15-auth.json
{
  "id": "thread-2024-03-15-auth",
  "features": ["F-001", "F-002"],
  "status": "active",
  "decisions": ["Use JWT over sessions"],
  "openQuestions": ["Token refresh strategy?"],
  "branchedFrom": null
}

Thread Branching

▼

When a brainstorm session diverges from an earlier thread, CAP creates a branch — preserving both the original direction and the new one. This is critical for tracking divergent ideas:

thread-2024-03-15-auth          (original: JWT approach)
  +-- thread-2024-03-18-auth-v2 (branch: session cookies instead)

Revisit Detection

▼

When starting a new brainstorm, CAP checks existing threads for keyword overlap and feature ID overlap. If a match is found, it surfaces the prior thread so you can resume, branch, or supersede rather than starting from scratch.

Multi-Developer: Git-Tracked Thread Index

▼

The thread-index.json is designed for git merge-friendliness. Each thread has a unique ID and anchor, so concurrent additions by different developers merge cleanly.

Thread Reconnection F-032

When revisiting a previous thread, the Synthesis Engine offers 4 strategies:

Merge
Combine old and new ideas into a single unified thread. Both perspectives preserved.
Supersede
New thread replaces old. Original archived with "superseded-by" link.
Branch
Fork into parallel exploration. Original remains active alongside new direction.
Resume
Continue the original thread. New entries appended, no structural changes.

AC Conflict Detection

▼

When merging or superseding, the engine performs a side-by-side comparison of acceptance criteria. Conflicts are flagged for manual resolution:

CONFLICT in F-001 AC #3:
  Old: "JWT tokens expire after 24 hours"
  New: "JWT tokens expire after 1 hour with refresh"

  Resolution required: merge | keep-old | keep-new

All resolutions are logged in the thread for audit trail.

Impact Analysis F-033

Proactive analysis during brainstorming that detects feature overlap, dependency chains, and circular dependencies.

Overlap Detection

When two features touch the same files or acceptance criteria, impact analysis flags the overlap before you start building — preventing wasted work.

Dependency Chain Traversal

Full transitive dependency analysis: if F-005 depends on F-003, which depends on F-001, you see the entire chain and can plan accordingly.

Circular Dependency Warning

Detects cycles in feature dependencies (F-001 → F-003 → F-007 → F-001) and proposes resolution strategies.

Impact analysis is advisory only — it surfaces information and proposes resolutions, but never modifies code or Feature Map entries automatically.

Memory Graph F-034

A connected graph structure stored in .cap/memory/graph.json that links features, threads, decisions, pitfalls, patterns, and hotspots.

6 Node Types

feature thread decision pitfall pattern hotspot

6 Edge Types

depends_on supersedes conflicts_with branched_from informed_by relates_to

Temporal Queries

▼

Every node and edge carries a timestamp. This enables temporal queries:

  • "What decisions were made about F-001 last week?"
  • "Which pitfalls were added since v2.3?"
  • "Show me the evolution of F-012 from planned to shipped"

Flat Files as Generated Views

▼

The graph is the canonical store. Flat markdown files (decisions.md, hotspots.md, pitfalls.md, patterns.md) are generated views — human-readable projections of the graph for quick scanning.

.cap/memory/
  graph.json        # Canonical store (nodes + edges)
  decisions.md      # Generated view
  hotspots.md       # Generated view
  pitfalls.md       # Generated view
  patterns.md       # Generated view
  threads/          # Thread files
    thread-index.json

Merge-Friendly Serialization

▼

The JSON serialization is designed for git merge friendliness:

  • Each node has a stable anchor ID derived from content (not auto-increment)
  • Nodes and edges are stored as sorted arrays (deterministic ordering)
  • Adding a node is an append operation — no rewriting existing entries
  • Most merges resolve automatically via anchor-ID deduplication

Multi-Developer Workflow

Memory is designed to be shared via git. Here is the typical multi-developer flow:

Developer A
# 1. Bootstrap memory from their sessions + code tags
/cap:memory init

# 2. Memory files generated in .cap/memory/
git add .cap/memory/
git commit -m "chore: bootstrap project memory"
git push
Developer B
# 1. Pull Developer A's memory
git pull

# 2. Run memory init — MERGES with existing memory
#    Anchor-ID dedup prevents duplicates
/cap:memory init

# 3. Both perspectives now combined
git add .cap/memory/
git commit -m "chore: merge Developer B memory"
git push

Because anchor IDs are content-derived, the same decision found by both developers produces the same anchor — it is deduplicated automatically, not doubled.

07Agents

5 specialized agents, each with a single responsibility. Communication only via shared artifacts.

B
cap-brainstormer

Conversational feature discovery. Guides the developer through questions, produces Feature Map entries with acceptance criteria, feature grouping, and dependency analysis.

P
cap-prototyper

Builds annotated code. 4 modes: prototype (new feature), iterate (refine existing), architecture (system design), annotate (tag existing code).

T
cap-tester

RED-GREEN discipline with adversarial mindset. Writes tests against Feature Map acceptance criteria. Tests must fail first (RED), then pass (GREEN).

R
cap-reviewer

Two-stage review. Stage 1: AC verification (does the code satisfy acceptance criteria?). Stage 2: Code quality (only runs if Stage 1 passes).

D
cap-debugger

Scientific method debugging. Persistent state across context resets. Hypothesis → experiment → observe → conclude. Deploy-aware workflow minimizes deploy cycles.

08Test Quality Infrastructure

Beyond coverage: assertion density, mutation testing, and a trust score.

Trust Model

▼

The /cap:test-audit command produces a composite trust score based on:

MetricWeightWhat it measures
Line coverage25%Basic coverage via c8
Assertion density25%Assertions per test — flags empty tests
Mutation score30%% of code mutations caught by tests
Anti-pattern score20%Absence of test smells (no mocks of SUT, no sleep, etc.)

Mutation Testing

▼

Mutation testing modifies your source code (mutants) and checks whether your tests catch the change. This reveals tests that pass but do not actually verify behavior.

// Original code
if (user.age >= 18) { allowAccess(); }

// Mutant: boundary change
if (user.age > 18) { allowAccess(); }
// If tests still pass, the boundary isn't tested!

Spot Checks

▼

Spot checks are lightweight, targeted verifications that run as part of the audit. They check for common testing anti-patterns:

  • Tests without assertions
  • Mocking the system under test
  • Tests that depend on execution order
  • Hardcoded timeouts/sleeps
  • Tests that always pass (tautological assertions)

09Monorepo Support

Automatic workspace detection and per-app scoping.

Workspace Detection

CAP detects monorepo workspaces automatically from package.json workspaces, pnpm-workspace.yaml, or directory conventions. Each workspace package gets its own tag namespace.

my-monorepo/
  packages/
    api/         # /cap:switch-app api
    web/         # /cap:switch-app web
    shared/      # /cap:switch-app shared
  FEATURE-MAP.md # Shared across all apps
  .cap/
    SESSION.json # Tracks active app

Per-App Scoping

When you run /cap:switch-app web, all subsequent commands scope to that package:

  • /cap:scan only scans files within packages/web/
  • /cap:prototype creates files within the active app
  • /cap:test runs tests for the active app only
  • The Feature Map remains shared — features can span apps

10Security

Zero dependencies, provenance tracking, and safe CI integration.

Zero Runtime Dependencies

CAP uses only Node.js built-in modules (fs, path, crypto). No external npm packages at runtime means zero supply-chain attack surface.

Provenance

Published with npm provenance attestation. Every published version is verifiably linked to its source commit.

Safe CI

CAP commands are designed to be CI-safe. No network calls, no file system operations outside the project root, no environment variable exfiltration.

11Migration from GSD

Upgrading from GSD Code-First v1.x? One command handles it.

/cap:migrate

Tag Conversion

GSD v1.x TagCAP v2.x Tag
@gsd-feature@cap-feature
@gsd-todo@cap-todo
@gsd-risk@cap-risk
@gsd-decision@cap-decision

What /cap:migrate Does

  • Scans all source files for @gsd-* tags and replaces them with @cap-* equivalents
  • Converts .planning/ directory structure to .cap/
  • Migrates session format from GSD v1.x to CAP v2.x
  • Preserves all feature IDs and acceptance criteria
  • Generates a migration report showing what changed
