The CAP Pipeline

Six phases, linear by default, re-entrant by design. Every phase feeds the next, every artifact is traceable.

Phase 1: Project Setup

1

Initialize Project

/cap:init

When: Once per project, or when joining an existing project.

Creates .cap/ structure, fetches library docs via Context7, detects monorepo, performs brownfield analysis, builds memory graph from existing data, and migrates past brainstorm sessions to conversation threads.

Best practice: Run even on existing CAP projects — it's idempotent and picks up new dependencies.

Phase 2: Feature Discovery

2

Conversational Feature Design

/cap:brainstorm [--resume] [--multi]

When: Starting new work, exploring a feature area, or when requirements are unclear.

The brainstormer agent asks one question at a time, references prior threads, and produces structured Feature Map entries with numbered acceptance criteria.

Nothing is written until you explicitly approve.

Phase 3: Build

3

Code-First Development

/cap:prototype --features F-001

When: After brainstorm, when features have ACs in the Feature Map.

Builds working, annotated code with @cap-feature and @cap-todo tags. Auto-runs /cap:scan to sync tag state back to Feature Map.

ModeFlagUse case
prototype(default)Build new feature from scratch
iterate--iterateRefine existing prototype
architecture--architectureDesign structure without implementation
annotate--annotateTag existing code retroactively

Phase 4: Refine

4

Close the Gaps

/cap:iterate

When: After initial prototype, to fill in missing acceptance criteria.

Scans for uncovered ACs, spawns prototyper in iterate mode, re-scans, repeats until all ACs are satisfied.

Best practice: Run before testing — catches missing ACs that would fail tests.

Phase 5: Test

5

Adversarial RED-GREEN Testing

/cap:test --features F-001 [--deep]

When: After prototype/iterate, when code is ready for verification.

RED Write tests that SHOULD fail if the AC isn't implemented.
GREEN Run tests against actual code, verify they pass.
DEEP Run test audit: assertion density, coverage, mutations, anti-patterns, trust score.

FlagEffect
--features F-001Scope to specific features
--red-onlyStop after RED phase (TDD workflow)
--deepRun full test audit after GREEN. Trust score < 70% triggers improvement suggestions.

Trust Score Breakdown

ComponentMax PointsWhat it measures
Assertion density30Assertions per test (2+ = full marks)
Code coverage30Lines (15), branches (10), functions (5)
Mutation score25% of code mutations caught by tests
Anti-patterns-15Penalty for weak assertions, snapshot abuse, etc.
Empty tests-10Penalty for tests with 0 assertions

When trust score is below 70%

The audit generates prioritized improvement suggestions sorted by potential point gain:

IMPROVEMENT SUGGESTIONS
Current score: 52/100 (target: 70+)

1. Increase assertion density (+20 pts)
Tests average 0.8 assertions each. Add specific value checks...

2. Fix 3 empty test(s) (+6 pts)
tests/core.test.cjs:42, tests/auth.test.cjs:15...

3. Fix 5 anti-pattern(s) (+10 pts)
Top issue: Weak assertion: only checks value is defined...

Claude will ask: "Trust score is 52/100. Want me to address the top suggestion now?" — and iteratively fix issues until the score reaches 70+.

Phase 6: Review

6

Two-Stage Quality Gate

/cap:review

When: Before merging, after tests pass.

Stage 1: AC compliance — every acceptance criterion verified against code.
Stage 2: Code quality — security, patterns, performance. Only runs if Stage 1 passes.

Tag System

Tags in source code are the traceability link between Feature Map and implementation.

// Source Code Feature Map
@cap-feature(feature:F-001) ↔ F-001: Implement Auth
  @cap-todo(ref:F-001:AC-1) ↔ AC-1: Login endpoint
  @cap-todo(ref:F-001:AC-2) ↔ AC-2: JWT tokens
  @cap-todo risk: session... → .cap/memory/pitfalls.md
  @cap-todo decision: bcrypt → .cap/memory/decisions.md
TagPurposeRequired?
@cap-featureMarks file as belonging to a featureYes
@cap-todo(ref:)Marks implementation of a specific ACYes
@cap-todo risk:Known risk or concernOptional
@cap-todo decision:Architectural decision with rationaleOptional

Daily Workflow

Morning

/cap:start # Restore session state

New Feature

/cap:brainstorm # Discover and define
/cap:prototype F-042 # Build it
/cap:iterate # Fill gaps
/cap:test F-042 --deep # Verify ACs + audit trust score
/cap:review # Quality gate
git push # Ship it

Bug Found

/cap:debug # Hypothesize, test, conclude

Joining Existing Project

/cap:init # Setup + detect stack + build memory
/cap:status # See where things stand
/cap:annotate # Tag existing code

Core Principles

1
Code is the plan
Build first, extract structure from annotated code. No upfront document-heavy planning.
2
Feature Map is the single source of truth
Not PRDs, not Jira, not Confluence. One file, human-readable, git-diffable.
3
Tags create traceability
Every line of code links back to a requirement. Scan extracts truth from code.
4
Agents are stateless
All state lives in Feature Map, SESSION.json, and code tags. Agents read and produce artifacts.
5
Nothing is written without approval
Brainstorm and review have explicit confirmation gates. You are always in control.
6
Memory persists across sessions
Decisions, pitfalls, and conversation threads survive context resets and compression.
