CAP — Code as Plan

Status: 2026-05-07 81 Features 59 shipped · 21 tested · 1 prototyped ~7100 Tests 22 Commands · 6 Agents · 60+ Lib-Module

Was du in diesem Dokument findest: Vollständiger Status-Snapshot von CAP — Architektur, alle Memory- und Self-Learning-Bricks, Workflows, Best Practices, Anti-Patterns, plus eine konkrete Schritt-für-Schritt-Migrationsanleitung für bestehende Projekte (am Beispiel GoetzeInvest).

1 · Was ist CAP?

CAP — Code as Plan ist ein Developer-Framework für Claude Code, das die Code-First-Philosophie umsetzt. Statt Document-Heavy Planning vorne dranzustellen, wird zuerst Code gebaut. Strukturiertes Planning entsteht aus dem annotierten Code, nicht davor.

Kern-These

Die Codebase ist die Spec. Tags wie @cap-feature und @cap-todo verlinken die Implementierung mit der FEATURE-MAP.md. Jede Architektur-Entscheidung lebt als @cap-decision direkt am Ort der Wahrheit.

Aligned mit Dave Farleys Modern Software Engineering — optimiere für Lernen und Komplexitäts-Management, nicht für Vollständigkeit der Anforderungs-Dokumentation.

Constraints

  • Tech-Stack: Pure JavaScript/Node.js (CJS), Markdown, JSON
  • Distribution: npx code-as-plan@latest
  • Zero runtime deps: Keine externen npm-Packages zur Laufzeit
  • Node.js: >=20.0.0

4 · Standard-Workflow

Fünf Schritte, linear by default, re-entrant by design:

┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ brainstorm │→ │ prototype │→ │ iterate │→ │ test │→ │ review │ │ │ │ │ │ │ │ │ │ │ │ ACs locked │ │ tags + │ │ refine │ │ adversarial │ │ stage 1+2 │ │ in F-Map │ │ proto code │ │ + memory │ │ pinning │ │ → ship/iter │ └──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘ │ ┌─────────────────────────────────────────────────┘ ▼ ┌──────────┐ iterate-then-ship loop │ iterate │ ◄────── bei findings (häufig) └──────────┘

Re-entry-Punkte

  • Mid-prototype hat Spec-Lücke? → @cap-decision(spec-gap) + weitermachen, nicht zurück zu brainstorm
  • Stage-2 findet foot-gun? → iterate mit konkretem Briefing → re-review
  • Real-world canary scheitert? → siehe F-082 em-dash: post-merge fix als micro-PR
  • Coverage-Gap? → cap:test-audit + zusätzliche adversarial tests

5.3 F-078 — Platform-Bucket

Cross-cutting Decisions, die nicht zu einem Feature gehören.

Layout

.cap/memory/
├── F-079.md                              ← per-feature
├── F-080.md                              ← per-feature
├── platform/
│   ├── observability.md                  ← topic-bucket
│   ├── security.md                       ← topic-bucket
│   ├── snapshots-unassigned.md           ← F-079 AC-6 fallback
│   └── checklists/
│       ├── auth.md                       ← subsystem-checklist
│       └── deployment.md                 ← subsystem-checklist

Promotion-Contract: explicit-only

Eine Decision landet NUR im platform-bucket, wenn sie EXPLICIT mit @cap-decision platform:<topic> getaggt ist:

// @cap-decision platform:observability
// Token telemetry uses session-scoped sampling at 1% to avoid
// flooding the analytics pipeline. Auto-promotion would scatter
// this across F-061 + F-070 + F-071 — keep it cross-cutting.

Plain @cap-decision bleibt im per-feature file. Beide gleichzeitig (feature:F-070, platform:observability) → Loud Parse-Error.

Extends-Resolver (AC-3)

Per-feature files können platform-topics referenzieren:

---
feature: F-070
extends: platform/observability
---

Reader resolved die Kette in einer lookup-pass. Cycle-Detection mit voller Pfad-Anzeige im Error-Message:

Cycle detected: F-070 → platform/observability → platform/telemetry → platform/observability

6 · Self-Learning Stack (V5)

Geschlossener Lernkreislauf — CAP lernt aus seinen eigenen Sessions, was funktioniert hat und was nicht. Sechs Bricks bilden zusammen die end-to-end-Loop.

┌─────────────────────────────────────────────────────────────────────┐ │ │ │ F-061 Token-Telemetrie ───► F-070 Signal-Sammlung │ │ (per session) (token, latency, errors, │ │ patterns-applied, success-rate) │ │ │ │ │ F-071 Pattern-Extraction ◄────────────┘ │ │ (heuristic + LLM via Skill-Briefing) │ │ │ │ │ ▼ │ │ F-072 Two-Layer Fitness Score │ │ (statistical + semantic, 3 deterministic probes) │ │ │ │ │ ▼ │ │ F-074 Auto-Retract ──► F-073 Pattern Review (cap:learn) │ │ (atomic-write, (manual gates for patterns above │ │ defense-in-depth) confidence threshold) │ │ │ └─────────────────────────────────────────────────────────────────────┘

6.1 F-061 — Token Telemetry

Observability-Fundament. Per-session token-counts, agent-spawn-times, tool-call-rates, error-rates. Ohne Telemetry kein Signal → kein Pattern.

6.2 F-070 — Collect Learning Signals

Subprocess-Hook-State via persistent ledger. Signals werden in .cap/learning/signals.jsonl persistiert. E2E spawnSync-Tests sind Pflicht — pure unit-tests entdecken hook-state-leaks nicht.

6.3 F-071 — Pattern Extraction

Heuristik-Layer + LLM-Layer. Wichtig: LLM via Skill-Briefing-Pattern (kein SDK-Call). Stage-2-Review fängt foot-guns die der Prototyp akzeptiert hat — V6-class-lesson #0.

6.4 F-072 — Fitness Score

Two-Layer-Score: statistical (success-rate über N runs) + semantic (LLM-judge: passt das pattern zur situation). 3-Layer-Determinismus-Probes: gleiche inputs → gleiche outputs (alphabetic sort-locks via @cap-risk-Pattern).

6.5 F-073 — Pattern Review (cap:learn)

Manual-Gate für Patterns über confidence-threshold. User reviewed pattern, kann promoten / reject / defer. Promoted patterns landen in .cap/learning/applied-state.json und werden vom prototyper at session-start gelesen.

6.6 F-074 — Pattern Unlearn & Auto-Retract

Wenn ein promoted pattern in 3 consecutive sessions zu fitness-drop führt → auto-retract. Defense-in-depth bei git-modulen, atomic-write für runtime-config, sandbox-discipline pinned.

7 · V6.1 — Monorepo + Multi-Format

Real-world readiness: F-081 + F-082 ermöglichen CAP in heterogenen Codebases.

F-081 — Multi-Format Parser

  • Long-form IDs: F-HUB-AUTH, F-PAYMENT-CHECKOUT (Union-Regex /^F-(\d{3,}|[A-Z][A-Z0-9_-]*)$/)
  • Bullet-style ACs: - [ ] AC-1: Description als Alternative zu pipe-tables
  • Format-Detection: Heuristic-first + explicit override via .cap/config.json:featureMapStyle
  • Safe-Mode: {safe: true} opt-in macht parser non-throwing bei duplicates (returns {features, parseError})

F-082 — Monorepo-Aggregation

  • Hard contract: ### Rescoped Feature Maps-Header listet sub-app paths
  • Opt-in directory walk: featureMaps.discover: "auto" | "table-only"
  • Header-Separator-Tolerance: Colon, em-dash, en-dash, hyphen-with-whitespace
  • Cross-sub-app-Resilience: parseError in einem sub-app blockt nicht die anderen

F-083 — File Extraction

Refactor: cap-feature-map.cjs 2336 → 1495 LOC, monorepo-Concerns in cap-feature-map-monorepo.cjs (882 LOC). Lazy-require Cycle-Resolution mit Identity-Preservation. Zero behavior change.

8 · Quality-Gates & Stage-2-Review

Jedes V6-Class-Feature durchläuft den gleichen Pipeline:

StageWerWas

2. Prototypecap-prototyper PROTOTYPECode + Tags + happy/adversarial tests

Die 12 V6-Class-Lessons (apply UPFRONT, nicht warten auf review)

  1. Proto-pollution defense — frontmatter, JSON-parse, Cache-Reads
  2. ANSI-injection defense — JEDER user-controlled interpolation-site (nicht nur main)
  3. Path-traversal — slug-regex, NUL-bytes, ../etc/passwd
  4. Silent-skip ist REAL silent — capture stdout+stderr, debug to file not console
  5. Cache TOCTOU benign + documented (never silent)
  6. Atomic writes via tmp+rename
  7. Round-trip stability — write → read → write byte-identisch
  8. Surface-Limit HARD-cap (nicht best-effort)
  9. Priority tie-break determinism (lexical or mtime)
  10. Missing/empty/malformed input graceful handling
  11. Realistic fixtures — synthetic-only-blindspot lesson aus F-082-iter2
  12. Read-only contracts in tests pinned (mtime+content+size unchanged)
Stage-2-Streak: 7-for-7 F-076, F-078, F-079, F-081, F-082, F-082-iter2 hatten alle 3+ findings. F-080 war das erste 0-finding-Feature, weil alle 12 Lessons UPFRONT applied wurden. Die Klasse hat eine 100%-Hit-Rate auf Foot-Guns die der Prototyper akzeptiert.

9 · Best Practices

1. Code-First, immer

Build first. Wenn du dich beim Schreiben von Spec-Documenten ertappst — wechsel zu Code mit Tags. Tags sind die Spec.

2. Stage-2 ist nicht optional

Auch bei "trivialen" Features. Die 7-for-7-Streak zeigt: prototyper sieht foot-guns nicht, die ein independent reviewer sofort erkennt.

3. Real-world canary

Synthetic fixtures haben blindspots. Mindestens ein Smoke-Test gegen echte Daten (z.B. Live-Repo dry-run) bevor ein V6-Feature shipped.

4. Memory-driven decisions

Bei jedem nicht-trivialen Choice: memory durchsuchen ob ein platform-topic oder per-feature-decision das schon entschieden hat.

5. Atomic writes

Tempfile + rename. Nie in-place. Defense-in-depth bei git-Module (F-074-Lesson). cap-memory-migrate.cjs hat den canonical helper.

6. Diagnose empirisch verifizieren

Issue-body ist Hypothese, nicht Fakt. CI #42-Lesson: 4× Path 1 abgelehnt, weil 4× empirisch reproduziert. Nicht raten — messen.

7. Spec-gap dokumentieren

Wenn der Prototyper mid-flow eine Lücke sieht: @cap-decision(F-NNN/spec-gap) reasoning — nicht zurück zu brainstorm.

8. Snapshot vor /compact

Jede non-trivial Session endet mit cap:save. Cross-session-continuity ohne compression-loss. Auto-linkage zu activeFeature.

Git-Workflow

  • Feature-Branches für neue Features (feature/F-079-snapshot-linkage)
  • Direct main für small fixes (one-line CI bumps, regex fixes)
  • Squash-merge für PRs — feat(F-079): wire snapshot linkage to features and platform (#49)
  • Manual Tests am Gruppen-Ende, nicht pro Feature (V5-Präferenz)
  • Pre-Push: npm run test:coverage lokal — nicht nur npm test

Non-blocking, max 1× pro Session. Suppressible via .cap/config.json:

Atomic per Stage

Falls eine Stage fehlschlägt (z.B. Stage 3 annotate timeout), wird sie isoliert geloggt — nachfolgende Stages laufen weiter. Nach dem Fehler einfach /cap:upgrade erneut aufrufen, der Marker zeigt die noch-zu-machenden Stages, der Rest läuft idempotent durch.

Wann manuell statt /cap:upgrade? Wenn du custom-tuning brauchst — z.B. nur einzelne Migrations-Schritte selektiv anwenden, oder zwischendrin manuelle Reviews brauchst. Sonst /cap:upgrade verwenden, ist robuster (atomic-Pipeline + Version-Marker + audit-log).
